> 规范、管理、展示都应 **在 demo 仓库里长出来**，而不是另建一套「大而全」的空目录。

## 三、分阶段拓展（建议顺序）

### Phase 1 — 现在：能管、能看、能演示 ✅ 进行中

| 做什么 | 落在哪 |
|--------|--------|
| Agent 登记格式 | `agents/*/agent.yaml` |
| 登记流程 | `社区demo/docs/agent-sop.md` |
| 目录展示 | `web/` |
| 构建发布 | `tools/build_github_dist.py` |

**出口**：公网链接 + 能给老师/平台演示。

---

### Phase 1.5 — 规范变硬：能校验、能验收

老师要的不只是「有 yaml」，还要 **好不好有依据**。

| 拓展项 | 做法 | 产出 |
|--------|------|------|
| 格式校验 | `validate.py` + schema | 错的 yaml 提交前就报错 |
| 评价字段 | 在 yaml 里用 `evaluation`（已有雏形） | 详情页展示验收指标 |
| 流程编排 | yaml 增加 `workflow.steps`（待讨论） | 写清步骤、报错策略，不只列 skill |
| 基准模板 | `docs/templates/baseline-agent.yaml` | 老师说的「基准 Agent + 领域 diff」 |

**原则**：新字段先写进 yaml 和 SOP，页面能展示多少算多少，不追求一次做满。

---

### Phase 2 — 评判与打磨：肿瘤 Agent 当第一个案例

| 做什么 | 说明 |
|--------|------|
| 试用诊断 | 同学测平台肿瘤 Agent，填问题表（任务拆分？上下文丢失？） |
| 问题入库 | 每条问题对应：Agent id、步骤、复现步骤、日志 |
| 页面体现 | 详情页加「已知问题 / 可复现性说明」（可先手工写 yaml） |
| 组内讨论 | 诊断结果 → 反推 benchmark 模板、skill_chain 该怎么改 |

**这是回应老师「评判、打磨、自进化」的最低成本入口**：一个真实烂 Agent 比十页空方案有用。

---

### Phase 3 — 与平台合体

| 做什么 | 说明 |
|--------|------|
| 「前往使用」 | 详情页跳 BioIntelOS 对应 Agent |
| 反馈 | 占位按钮接平台接口 |
| 数据来源 | yaml 手工维护 → 平台 API 拉清单（像题库从 seed 迁到数据库） |
